# VideoRAG-for-EdgeAI

Fully local, multimodal video question-answering pipeline built to run on a Raspberry Pi 5 (4GB RAM). No cloud APIs, no `ultralytics` dependency — everything runs on-device.

Ask natural language questions about a video ("what color was the car?", "what does the sign say?", "summarize what happened") and get timestamp-grounded answers generated by a local LLM.

## How it works

```
video.mp4
   ├── frames (sampled @ 1fps) ──► YOLOX-Nano (ONNX) ──► object labels
   │                           └─► Tesseract OCR (every Nth frame) ──► on-screen text
   │                           └─► MobileNetV3 (ONNX, optional) ──► visual embedding
   └── audio track ──► faster-whisper (tiny/base) ──► timestamped transcript

All of the above → short timestamped text documents
                 → embedded with BGE-small-en-v1.5
                 → stored in ChromaDB (local, on-disk)

query (text) → embed → retrieve top-k timestamped docs → Ollama (local LLM)
            → grounded, timestamp-cited answer
```

**Why these components:**
- **YOLOX-Nano** instead of `ultralytics`/YOLO — Apache-2.0 licensed, ~7MB, ONNX-native, no extra package dependency.
- **faster-whisper** instead of `openai-whisper` — CTranslate2 backend with int8 quantization, much lower RAM/CPU on ARM.
- **BGE-small** text embeddings — small footprint, solid quality for RAG over short captions/transcripts.
- **ChromaDB** — embedded vector store, no separate server, persists to disk.
- **Ollama** for generation — decoupled from the rest of the pipeline, so it can run on the Pi or be pointed at a bigger model on another machine on your LAN.


## Requirements

- Raspberry Pi 5 (4GB+ RAM) or any Linux machine
- Python 3.10+
- `ffmpeg`, `tesseract-ocr`
- [Ollama](https://ollama.com) installed and a small local model pulled (e.g. `qwen2.5:3b`)

## Setup

```bash
git clone <this-repo-url>
cd video_rag_pi
chmod +x setup.sh download_models.sh
./setup.sh
```

`setup.sh` installs system packages, creates a Python venv, installs dependencies, and downloads the ONNX model weights (YOLOX-Nano, MobileNetV3). It assumes a Debian-based OS (Raspberry Pi OS) by default — edit the package manager lines near the top if you're on a different distro (e.g. `pacman` on Arch Linux ARM).

Pull a local generation model:
```bash
ollama pull qwen2.5:3b
```

If either ONNX weight URL in `download_models.sh` has moved, it prints manual fallback links — grab the file and place it at the path shown, matching `config.py`.

## Usage

```bash
source .venv/bin/activate

# Index one or more videos
python main.py ingest data/videos/clip1.mp4 data/videos/clip2.mp4

# Ask questions
python main.py query "how many people appear in the video?"
python main.py query "what does the sign in the shop say?" --video clip1
```

Each answer is generated from retrieved, timestamped evidence, and the model is prompted to cite timestamps. The retrieved evidence lines are printed above the answer so you can see exactly what it was grounded on.

### Notebook version

`video_rag_pipeline.ipynb` walks through the same pipeline interactively, cell by cell — useful for inspecting intermediate outputs (detections, OCR text, transcript segments) while developing or debugging on a new video.

## Tuning for low-RAM devices

All settings live in `config.py`:

| Setting | Effect |
|---|---|
| `SAMPLE_FPS` | Lower = fewer frames processed = less RAM/CPU. Try 0.5 for long videos. |
| `ENABLE_VISUAL_EMBEDDINGS` | Set `False` to skip MobileNetV3 entirely if you only need text-grounded retrieval. |
| `ENABLE_OCR` / `OCR_EVERY_N_FRAMES` | Tesseract is CPU-heavy; raise the interval or disable if you don't need on-screen text. |
| `WHISPER_MODEL_SIZE` | `tiny` uses far less RAM than `base` — drop to `tiny` first if you hit OOM. |
| `MAX_FRAME_DIM` | Frames are downscaled before detection; lower this to cut detector cost further. |

The pipeline processes videos one at a time, sequentially, and loads all models once per `ingest`/`query` invocation rather than per frame, to keep peak RAM predictable. On a 4GB Pi, prefer the CLI over the notebook for actual ingestion runs — the notebook keeps all models resident across cells, which uses more memory than the script's clean start/exit per command.

If you hit `OSError: [Errno 28] No space left on device` or Ollama's `no space left on device` during a model pull, see [Storage notes](#storage-notes) below.

## Storage notes

Model weights (Ollama models, HuggingFace caches, pip cache) add up fast relative to a typical Pi SD card. If you're running low on space:

```bash
# clear caches
pip cache purge
sudo apt clean

# move Ollama's model storage to a larger external drive
sudo systemctl stop ollama
sudo mkdir -p /mnt/ssd/ollama-models
sudo mv /usr/share/ollama/.ollama/models/* /mnt/ssd/ollama-models/
sudo chown -R ollama:ollama /mnt/ssd/ollama-models
sudo systemctl edit ollama
# add under [Service]:
#   Environment="OLLAMA_MODELS=/mnt/ssd/ollama-models"
sudo systemctl daemon-reload
sudo systemctl start ollama
```

Consider also relocating the project's Python venv and HuggingFace/pip caches to the same external drive if space stays tight.

## Extending

- Swap `qwen2.5:3b` for `phi3:mini`, `llama3.2:3b`, or any other Ollama-servable model in `config.py`.
- Visual embeddings (if enabled) are stored per-frame in ChromaDB metadata — add an image-to-image query mode by embedding a query frame with `MobileNetEmbedder` and running a vector search against those, mirroring the text retriever pattern.
- Route queries through multiple retrieval strategies (temporal, spatial, counting, audio-specific) before the generation step if you want category-specific handling rather than one general-purpose retriever.
